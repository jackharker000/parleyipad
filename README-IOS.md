# Parley — native iPad app (Capacitor)

The iPad app is the same Parley code, shipped as a real installable iOS app
instead of a website. Everything latency-critical lives **on the device**:
the UI, IndexedDB (people, voiceprints, conversations, memories), the
AudioWorklet mic capture, and the ONNX VAD / speaker-ID models are all
bundled inside the app. Only the keyed AI/STT/TTS server functions go over
the network to the hosted deploy (https://parley.help), which holds the API
keys — exactly the trust boundary CLAUDE.md prescribes.

## How it fits together

| Piece | Where it runs |
| --- | --- |
| Cockpit UI, routing, IndexedDB, audio pipeline, VAD + speaker models | On the iPad, inside the app bundle |
| `createServerFn` calls (suggestions, summaries, STT tokens, TTS, …) | Hosted deploy (`VITE_API_ORIGIN`, default `https://parley.help`) |
| Supabase auth + per-account cloud backup | Direct from the device to Supabase (unchanged) |
| ElevenLabs streaming TTS WebSocket | Direct from the device (unchanged) |

Three small pieces make this work — all inert on the normal website:

1. **`CAP_BUILD=1` build mode** (`vite.config.ts`) — builds the client as a
   static SPA shell instead of SSR. The regular web build is untouched.
2. **`src/lib/native-bridge.ts`** — inside the Capacitor shell, rewrites
   server-function fetches (relative `/_serverFn/...` paths) to the hosted
   origin. No-op on the web.
3. **CORS in `src/server.ts`** — the hosted deploy answers preflights and
   adds `Access-Control-Allow-Origin` for `capacitor://localhost` only.

## Building the app (requires a Mac with Xcode 16+)

```bash
bun install
bun run build:ios     # CAP_BUILD=1 vite build → assemble dist-ios → cap sync ios
bun run open:ios      # opens ios/App in Xcode
```

Then in Xcode:

1. Select the **App** target → Signing & Capabilities → pick your Apple
   Development team (bundle id `help.parley.app` — change it there if your
   team can't register it).
2. Plug in the iPad, select it as the run destination, press **Run**.

That's it — Capacitor 8 uses Swift Package Manager, so there is no CocoaPods
step. For an App-Store/TestFlight build use Product → Archive as usual.

To point a build at a different backend (e.g. a preview deploy):

```bash
VITE_API_ORIGIN=https://parley-git-branch-xyz.vercel.app bun run build:ios
```

## Permissions the app requests

- **Microphone** — transcription + speaker identification (the core of the app).
- **Location (while using)** — recognising saved Places to pre-select the
  right people/context. Denying it just disables place detection.

Both usage strings live in `ios/App/App/Info.plist`.

## Notes & gotchas

- **iPad-only** by design (`TARGETED_DEVICE_FAMILY = 2`), portrait and
  landscape both supported.
- The web deploy must be live for AI features; without network the app still
  opens and quick phrases / cached TTS / typed-text speech keep working
  (graceful-degradation path, unchanged).
- `dist-ios/` and `ios/App/App/public/` are generated by `build:ios` and
  gitignored — never edit them by hand.
- After changing web code, re-run `bun run build:ios` and press Run in Xcode
  again; no other steps.
- WebGPU in WKWebView requires iOS 18+; on older iPadOS the ONNX runtime
  falls back to WASM (slower embeddings, same behaviour). Test speaker-ID
  latency on the real device early — this is the step-6 checklist item from
  CLAUDE.md.
